2. LUMEN

Lumen 是 UE5 的实时全局光照和反射系统，核心设计目标是打破“烘焙光照”与“实时光线追踪”之间的壁垒，在保证次世代画质的前提下，适配从主机到高端 PC 的广泛硬件。Lumen 的运作机制是以“表面缓存”为核心，结合“屏幕空间”与“距离场追踪”的混合管线。

核心机制：混合追踪管线

Lumen 并不是每一次都发射昂贵的光线，而是按照成本“由低到高”的顺序进行分层处理：

  • 第一层：屏幕空间 tracing

    • 原理：Lumen 首先尝试在屏幕空间（Screen Space）发射光线。如果光线击中的物体就在当前画面内，它直接复用屏幕上的几何和材质信息。
    • 作用：这是最快的一步，能有效处理细小的几何体细节，并弥补后续步骤中“Lumen 场景”与真实三角形场景之间的不匹配。
    • 局限：无法处理屏幕外（Off-screen）或被遮挡的物体。

  • 第二层：光线追踪

    • 当屏幕空间追踪失败（例如光线射向了屏幕外或被遮挡区域），Lumen 会启动真正的光线追踪。根据硬件能力，它会自动选择以下两种模式之一：

    • A. 软件光线追踪 (默认)

      • 核心：有向距离场 (Signed Distance Fields, SDF)。
      • 机制：Lumen 不直接追踪高面数的三角形网格，而是追踪其网格距离场 (Mesh Distance Fields)。
        • 近距离：使用独立的 Mesh Distance Fields，精度较高。
        • 远距离：使用合并后的全局距离场 (Global Distance Field)，速度极快但精度较低。
        • 优势：支持广泛硬件（仅需 Shader Model 5，如 GTX 1070+），无需专用 RT 核心。
        • 劣势：无法处理形变网格（骨骼动画），对植被和薄壁物体支持较差。

    • B. 硬件光线追踪

      • 核心技术：基于 BVH 的真实三角形追踪。
      • 机制：直接调用显卡的 RT 核心（如 NVIDIA RTX / AMD RX 6000+）追踪真实的几何体（或 Nanite 代理网格）。
      • 优势：精度最高，支持蒙皮网格（角色）、半透明材质和高质量镜面反射。
      • 代价：性能开销大，显存占用高。

关键加速技术：表面缓存 (Surface Cache)

为了避免每次光线击中物体都进行昂贵的材质计算，Lumen 引入了Surface Cache。

  • Card System：Lumen 会在物体表面自动生成一系列低分辨率的 2D 贴图（Cards）。
  • 引擎预先（或运行时摊销）捕获物体表面的反照率（Albedo）、自发光等属性，并计算直接光照和一次反弹的间接光，存储在这些 Cards 中。
  • 当光线击中物体时，Lumen 不会运行复杂的材质 Shader，而是直接去读取该位置 Surface Cache 中的光照数据。

反射处理

Lumen 的反射与 GI 紧密结合，同样采用了混合策略：

  • 优先级：首先使用屏幕空间反射 (SSR) 处理高频细节。
  • 回退：SSR 失败后，使用 Lumen 追踪（SDF 或 硬件 RT）。
  • 质量分级：
    • Surface Cache 模式：反射中的物体主要显示 Surface Cache 的结果，性能好但镜面反射较模糊。
    • Hit Lighting 模式（仅限硬件 RT）：在反射击中点运行真实的 Shader，可实现完美的镜面反射，但极其昂贵。

针对大型开放世界的设计

  • Lumen Scene 串流：Lumen 场景只在摄像机周围生成（默认约 200 m 范围），随着玩家移动动态更新。
  • 远场追踪 (Far Field)：结合世界分区（World Partition）和 HLOD，Lumen 支持长达 1 km 的远距离全局光照（主要用于硬件光线追踪模式下），解决了远景“漏光”或变黑的问题。

与 Nanite 的协同

Lumen 和 Nanite 是互补的。Nanite 负责处理微多边形几何体，而 Lumen 利用 Nanite 生成的代理网格（Proxy Mesh）来加速硬件光线追踪计算，避免了传统高模对光追性能的影响。

当前局限

  • 只有静态网格支持最好：软件光追模式下，不支持骨骼动画（如角色），角色只能依赖屏幕空间追踪或简化的胶囊体阴影。
  • 薄壁几何体问题：SDF 难以精确表示非常薄的物体（如单面墙、树叶），容易产生漏光（Light Leaking）。
  • 透明物体：Lumen 对半透明材质的支持有限，通常只能获得低质量的 GI。
  • 室内镜面反射：在不开启硬件光追的“Hit Lighting”模式下，镜子里的反射可能看起来比较模糊或有噪点。
  • VR 支持：虽然技术上可行，但由于 VR 对高帧率和高分辨率的苛刻要求，目前 Lumen 并不适合 VR 开发。

3. UE 的虚拟纹理原理 源码

核心概念

虚拟纹理（Virtual Texturing，MegaTexture / Sparse Virtual Texture）本质上是纹理内存管理的“虚拟化”技术。类似于操作系统的“虚拟内存”允许程序使用超过物理 RAM 大小的内存空间，虚拟纹理允许渲染引擎使用远超显存（VRAM）容量的巨大纹理（例如 16K 或更大），且只在运行时动态加载当前摄像机视角内可见的那部分纹理数据。

运作机制

虚拟纹理的实现依赖于将巨大的纹理切分为无数个固定大小的小块，称为 Page（页） 或 Tile（瓦片）（通常为 128x128 像素）。其渲染流程包含以下关键步骤：

  • 物理缓存 (Physical Cache / Pool)：
    显存中预分配的一块固定大小的纹理池。它不存储完整的纹理，只存储当前屏幕上可见的那些瓦片数据。
  • 页表 (Page Table / Indirection Texture)：
    这就好比一个“目录”。当 Shader 采样一个虚拟纹理时，它首先通过 UV 坐标查询这个极低分辨率的页表（页表通常是一张低分辨率的纹理，存储了当前虚拟页对应的物理页位置。）。页表会告诉 Shader：“你要找的这块纹理数据，现在位于物理缓存的哪个位置”。
  • 反馈分析 (Feedback Analysis)：
    这是一个 GPU 到 CPU 的反馈回路。每一帧渲染时，GPU 会记录下“当前画面需要哪些瓦片”写到一个专门的 Feedback Buffer。CPU 读取这个缓冲，分析出哪些瓦片是缺失的或精度不足的，然后从磁盘读取数据并上传到物理缓存中。

Unreal Engine 中的两种虚拟纹理

UE 将这一技术细分为两种截然不同的应用场景：

流送虚拟纹理 (Streaming Virtual Texturing - SVT)

  • 定位：替代传统的 Mipmap 纹理流送，主要用于处理超高分辨率的静态贴图。
  • 机制：纹理数据预先存在硬盘上。运行时，引擎根据反馈按需从硬盘读取瓦片到显存。
  • 优势：
    • 支持 UDIM(U-Dimension)：可以处理电影级的多象限 UV 贴图，打破了单张贴图最大 8K 的限制。
    • 显存节省：这是“按需付费”。对于一张 8K 贴图，如果物体在远处只占屏幕 100 像素，传统流送可能加载整个低级 Mip，而 SVT 只加载涉及到的那几个瓦片。
    • 光照贴图 (Lightmaps)：支持巨大的烘焙光照贴图而不撑爆显存。
  • 代价：增加了磁盘 I/O 压力；如果硬盘读取慢，快速转头时可能会看到短暂的模糊（Texture Popping）。

运行时虚拟纹理 (Runtime Virtual Texturing - RVT)

  • 定位：用于实时合成和复杂地形混合，主要优化渲染性能 (Pixel Shader Cost) 而非显存。
  • 机制：纹理数据不是来自硬盘，而是由 GPU 在运行时实时生成的。
    • 例如：你有一个极其复杂的 Landscape 材质（包含 10 层混合、噪波计算、程序化纹理），计算非常昂贵。
    • RVT 会将这些昂贵的计算结果“烘焙”（Cache）到一张虚拟纹理中。
    • 之后的每一帧，地形直接采样这张生成好的 RVT，跳过复杂的材质计算。
  • 典型应用：
    • 地形混合 (Landscape Blending)：让石头、草地与地形无缝融合（通过采样地形的 RVT 颜色和高度）。
    • 样条线贴花 (Spline Decals)：在地形上画道路，不需要产生额外的 Draw Call 或复杂的 Decal 投影，直接画进 RVT 里。
  • 特性：适合大面积、材质复杂但在运行时相对静止（或变化不频繁）的表面。

Pro Tips

  • 不要滥用：虚拟纹理采样比普通纹理采样更昂贵（因为需要先查页表，多了一次纹理读取）。如果你的贴图很小（如 1K 以下），用普通纹理性能更好。
  • RVT 的更新开销：虽然 RVT 采样很快，但写入（渲染）RVT 是有成本的。如果地形材质包含大量动态元素（如风吹动的波纹），导致 RVT 每帧都要重新渲染（Invalidate & Redraw），性能反而会下降。
  • 反馈延迟：由于 feedback buffer 需要回传 CPU 处理，纹理加载通常有几帧的延迟。在高速运动的游戏中，需要调优预加载策略以避免“马赛克”现象。

4. Nanite Virtualized Geometry

核心机制

Nanite 的本质是一套全新的网格数据格式和渲染管线，旨在解决实时渲染中多边形数量预算和 Draw Call 开销的瓶颈。Nanite 的本质是将几何体数据流送与GPU 驱动的细粒度剔除/光栅化相结合，将瓶颈从“多边形数量”转移到了“屏幕像素数量”和“硬盘 I/O 带宽”上。

A. 数据结构：簇与层级

  • Clustering：在导入模型时，Nanite 会将网格拆解为一个个微小的簇（Clusters），每个簇通常包含 128 个三角形。这是 Nanite 调度的基本单位。
  • Hierarchy：这些簇会被组织成一个类似 BVH（包围体层次结构）的树状结构。父节点是子节点簇的简化版本（LOD）。
  • Streaming：类似于纹理流送，Nanite 只会根据摄像机的距离和视角，将当前需要的几何体数据块流送到显存中。

B. 运行时选择：视点相关的 LOD (View-Dependent LOD)

  • 传统的 LOD 是基于对象的（整个模型在距离 X 切换到模型 B）。
  • Nanite 的 LOD 是**基于簇（Per-Cluster）**的。在每一帧，系统会计算屏幕误差（Screen Space Error），决定加载树结构中的哪一层级。
    • 这不仅实现了像素级别的细节精度，还彻底消除了传统 LOD 切换时的“Popping（突变）”现象。
    • 同一个物体的不同部分可以处于不同的 LOD 级别（例如：一根长柱子，靠近镜头的底端是高精度的，远离镜头的顶端是低精度的）。

C. 混合渲染管线：微多边形光栅化

这是 Nanite 性能的关键。传统的硬件光栅化器在处理小于一个像素的三角形（Pixel-sized triangles）时效率极低（Quad Overdraw 严重）。Nanite 实现了一个混合渲染器：

  • 软件光栅化（Software Rasterizer）：对于微小三角形（通常小于像素大小），Nanite 使用高度优化的 Compute Shader 在计算单元上进行光栅化。这比硬件管线快得多。
  • 硬件光栅化（Hardware Rasterizer）：对于较大的三角形，仍然交给 GPU 的固定光栅化单元处理。

背景：硬件“2x2”的光栅化

现代 GPU 的光栅化硬件（Raster Unit）并不是以“1 个像素”为单位工作的，而是以 2x2 的像素块（Quad） 为最小调度单位。

  • 为什么是 2x2？
    为了计算纹理采样的 Mipmap 层级，GPU 需要计算 UV 坐标相对于屏幕空间的导数（ddx/ddy）。这需要通过差分相邻像素的数据来获得。如果没有 2x2 的块，就无法计算导数。

  • 产生的问题（Quad Overdraw）
    当三角形非常小（例如：在一个 2x2 的 Quad 中，三角形只覆盖了其中 1 个像素中心）时：

    • 硬件必须启动全部 4 个像素线程（Pixel Shader Invocations）来处理这个 Quad。
    • 其中 1 个线程是有效的（写出颜色/深度）。
    • 另外 3 个线程是 Helper Lanes（辅助线程）。它们会执行所有的 Shader 计算以保证导数正确，但最后的结果会被丢弃。

  • Nanite 的目标是实现“一像素一三角形（One triangle per pixel）”。在这种极限情况下，意味着绝大多数三角形都非常小。如果是硬件光栅化：你会面临 75% 的算力浪费（每 4 个线程只有 1 个有用）。这在性能上是不可接受的。

解决方案：软件光栅化 (Software Rasterizer)

既然硬件管线对于微小三角形效率低，Epic 决定绕过硬件光栅化器，利用 Compute Shader 自己写了一套光栅化逻辑。

  • 工作原理：

    • Nanite 的软件光栅化器是针对**微多边形（Micropolygons）**特化的。
    • 它不依赖 2x2 Quad。因为它处理的是极小的几何体，通常不需要复杂的纹理导数计算（或者用其他方式近似）。
    • 它使用 Compute Shader 的线程直接计算几何体覆盖了哪些像素。
    • 它使用 64-bit Atomics（原子操作） 直接向显存中的一张名为 Visibility Buffer 的纹理写入数据（通常存的是深度值 + 三角形 ID）。

  • 性能优势：

    • 零浪费：如果一个三角形只占 1 个像素，Compute Shader 就只处理这 1 个像素的数据，没有额外的 Helper Lanes 开销。
    • 并行度极高：GPU 的通用计算单元（ALU）非常擅长处理这种大规模并行的数学测试（点是否在三角形内）。
    • 据 Epic 官方数据，在处理像素级三角形时，软件光栅化比硬件光栅化快 3 倍。

为什么还需要硬件光栅化？

  • 大三角形的劣势：
    如果一个三角形占据了屏幕上 1000 个像素，软件光栅化就需要写复杂的循环逻辑去遍历这些像素，这在 Compute Shader 里效率并不高（存在分支预测和负载均衡问题）。
  • 硬件的优势：
    GPU 里的**固定功能光栅化单元（Fixed-function Rasterizer）**是专门设计的专用电路，用来快速填充大面积的像素。对于大三角形，传统硬件管线依然是王者。

Nanite 的调度逻辑 (Cluster Classification)

Nanite 在运行时（每帧）会进行一次非常巧妙的分类：

  • 分析簇（Cluster）：系统计算每个几何体簇（Cluster）在屏幕上的视觉大小。
  • 分流：
    • 小三角形簇 送入 Compute Shader 队列（软件光栅化）。这是 Nanite 的主要负载。
    • 大三角形簇 送入 标准图形管线（硬件光栅化）。

D. 剔除（Occlusion Culling）

Nanite 执行极其细粒度的剔除。它是以“簇”为单位进行的，甚至在光栅化之前就能剔除被遮挡的几何体。这使得即使场景中有数亿个多边形，只要它们不可见，就不会产生着色开销。

关键特性与优势

A. 几何细节的“无限”预算

  • 直接导入高模：可以直接导入数百万面的 ZBrush 雕刻模型或 CAD 数据，无需手动拓扑或烘焙法线贴图。几何体细节是真实的，而非贴图模拟的。
  • 性能与场景复杂度解耦：Nanite 的渲染开销主要取决于屏幕分辨率，而不是场景中的多边形总数。无论场景放 1 个还是 1000 个高模，只要占据的屏幕像素一样，渲染成本就相差不大。

B. 虚拟阴影贴图 (Virtual Shadow Maps - VSM)

Nanite 与 VSM 是伴生技术。由于几何体细节极高，传统的 Shadow Map 精度不够且开销巨大。VSM 类似虚拟纹理，只缓存和更新当前屏幕可见部分的阴影页面，从而实现高质量、软边缘且高性能的阴影。

C. 支持的复杂材质与形变

早期的 Nanite 限制较多，但 5.6 文档指出现在的 Nanite 已支持：

  • 世界位置偏移 (WPO)：支持风吹草动等顶点动画（但有性能开销，且可能导致接缝）。
  • 遮罩材质 (Masked Materials)：支持镂空效果（如铁丝网、树叶），但比不透明材质开销大。
  • 曲面细分 (Tessellation)：Nanite 引入了可编程的位移，允许在运行时动态增加几何细节，而无需预先细分网格。

D. 代理网格 (Fallback Mesh)

尽管 Nanite 渲染高模，但物理引擎和某些光照计算（如 Lumen 的软件光追模式）跑不动高模。

  • Nanite 会自动生成一个粗糙的代理网格（Fallback Mesh）。
  • 物理碰撞使用这个粗糙网格。
  • Lumen 软件光追使用这个粗糙网格的距离场（SDF）来计算光照，而画面渲染使用 Nanite 高模，两者结合实现性能平衡。

局限性

虽然 Nanite 很强大，但并非适用于所有情况：

  • 半透明 (Translucency)：虽然已有支持，但半透明对象的排序和光照计算在延迟渲染管线中依然昂贵，通常不建议对由于大量层叠导致的复杂半透明物体使用 Nanite。
  • 数据体积：虽然省去了法线贴图，但数百万面的几何体数据本身会占用较大的硬盘空间。
  • 非刚体变形：虽然支持 WPO，但对于骨骼动画极其复杂的角色，传统的蒙皮管线目前可能仍优于 Nanite（尽管 Epic 正在推进 Nanite 对骨骼网格体的支持）。
